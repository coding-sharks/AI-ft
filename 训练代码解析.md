## 0. 一句话总览

**这是一个「流式音频交互大模型」(Audio Interaction Model / LAIM)，做法是：把 Qwen2.5-Omni 的音频编码器当作冻结的离线特征提取器，把 Qwen2.5-3B 语言模型做全参数微调；音频不进词表、而是以「连续特征直接覆盖占位 token 的 embedding」的方式注入；监督的 token 是「每个音频块该沉默还是该开口说话」的决策 + 助手回复文本。** 开源训练用 Lightning Fabric + FSDP 在 8 卡上跑。
>
> ⚠️ **注意**：这里的「交叉熵」是**开源代码**的写法。**论文真实目标是加权双损失 `L = L_LM + λ·L_stream`（λ=1.0）+ 四阶段课程训练**——开源版只是其 λ=1、单阶段的简化切片，详见第 8.5 节。

- **上游模型**：初始化自 **Qwen2.5-Omni-3B**（音频编码器和语言主干都来自它）。
- **训练框架**：Lightning **Fabric**（手写训练循环）+ 从 **litgpt 0.5.8** vendored 过来的 GPT 实现。
- **论文/品牌**：清华 Zhifei Xie 等，方法名 **SoundFlow**，arXiv:2606.05121（日期是未来的，疑似占位）。
- **模态方向**：**音频输入 → 文本输出**（含情绪标签），**不是** speech-to-speech（没有任何音频 codec 输出头）。

## 1. 项目在做什么（任务定义）

传统音频模型是「给一段音频 → 等 → 出一个回答」。本项目要做的是一个**常驻的「感知 → 决策 → 回应」循环**：模型持续吞入音频帧，**自己决定什么时候该开口**，默认保持 `⟨Silent⟩`（沉默/继续听），只在需要时才输出。（README.md:11-20）

它想用一个模型统一：
- **离线音频任务**：ASR、语音翻译(S2TT)、音频问答(AQA)；
- **实时流式任务**：语音对话、实时转写/翻译；
- **主动介入**：听到特定声音（如烟雾报警）时无需唤醒词主动开口。（README.md:204-208）

> 论文自称的效果（README 声明，非本地复现）：3B 模型在 MMAU 的 Audio 平均分 58.15，高于 Qwen2.5-Omni-7B 的 49.58；S2TT BLEU en→zh 55.22 / zh→en 35.21。

**训练方法学 SoundFlow 的三根支柱**（README.md:216-221，纯文字描述、无公式）：
1. 把短音频片段**拼接成长交互序列**（对应下面第 4 节的在线数据构造）；
2. **块级(chunk-level)的「说 / 不说」决策训练**，带「历史回看」和「理解感知式沉默」（对应 `KEEP_SILENCE` 监督）；
3. **异步 FIFO 推理**，把首帧延迟降低 4.5×（这是推理侧，不在训练代码里）。

代码里能确认的是支柱 1 和 2；README 明确标注完整数据集与完整训练 pipeline 为「Coming」（README.md:46-47）。

## 2. 训练全景图

```
 原始 JSONL (音频路径 + 助手文本[+情绪])
        │
        │  【离线一次性数据构造】cons_offline_data.py / cons_online_data.py
        ▼
 ┌────────────────────────────────────────────────────────────┐
 │ ① (在线)把多轮短音频 + 随机背景噪声静音 拼成一整条长波形          │
 │ ② 用 Qwen2.5-Omni audio_tower 编码整条音频 → AudioFeat.pt       │
 │    （特征已被 tower 的 proj 投影到 LM 隐藏维 n_embd）            │
 │ ③ 铺 token 序列：控制token + PAD/AUDIO_PAD 占位 + 文本           │
 │    同时生成 labels（非监督位置全部 = -100）                      │
 └────────────────────────────────────────────────────────────┘
        │  产出：训练就绪 JSONL {tasks, input_ids, labels, audio_pos, pt_path_dir}
        │        + 每条样本一个 AudioFeat.pt
        ▼
 ┌────────────────────────────────────────────────────────────┐
 │ dataloader.py：按字节偏移懒加载每行 → torch.load(.pt)          │
 │                → input_ids 用 PAD、labels 用 -100 右侧补齐到 4096 │
 └────────────────────────────────────────────────────────────┘
        ▼
 ┌────────────────────────────────────────────────────────────┐
 │ train.py（Lightning Fabric + FSDP，8×GPU）                    │
 │   model = GPT(Qwen2.5-3B)  ← 全参数可训练                      │
 │   forward: x = wte(idx)                                       │
 │            用 AudioFeat 覆盖 audio_pos 处 PAD 的 embedding      │
 │            → transformer blocks → lm_head                     │
 │   loss = 移位交叉熵(logits[:-1], labels[1:])，只算未被 mask 的位置 │
 │   AdamW lr=2e-4, wd=0.01, warmup200 + cosine, bf16, 2 epoch    │
 └────────────────────────────────────────────────────────────┘
        ▼
   out_dir/step-*/lit_model.pth  ─┬─ extract_state_dict.py（剥成纯权重）
                                  └─ wrap_audio_tower.py（把 adapter 塞回 tower.proj）
                                          → 推理用 checkpoint
```

**最关键的一点认知**：音频编码器**不参与训练**。它在数据构造阶段就被冻结、离线跑完，结果存成 `.pt`。训练时模型只是把这些**已经是 `n_embd` 维**的特征向量，直接写进占位 token 的 embedding 槽里。所以训练循环里没有任何卷积/whisper 前向，loss 也就是纯粹的语言模型交叉熵。

## 3. 模型结构与音频融合

### 3.1 主干
- 核心类 `class GPT(nn.Module)`（`model.py:43`），它**不是** import litgpt 的 GPT，而是把 litgpt 单文件模型 **vendored 复制过来就地改**的版本。配套 `Block / CausalSelfAttention / GptNeoxMLP / LLaMAMLP / GemmaMLP / LLaMAMoE / KVCache / RMSNorm` 都是 litgpt 标准件。
- 架构参数来自 `Config.from_file(checkpoint_dir/"model_config.yaml")`（`train.py:57`），`config_dir = ${CHECKPOINT_ROOT}/qwen-2.5-3B`。
- ⚠️ **真实维度本仓库里读不到**：`model_config.yaml` 不在仓库中；`config.py` 里的 `n_layer=16 / n_embd=4096 / vocab=50254` 是 litgpt 的**占位默认值，不是本模型的真实值**，不要引用。

  > 参考（来自公开 Qwen2.5-3B，非本仓库读出，仅供心里有数）：hidden ≈ 2048、约 36 层、GQA（16 注意力头 / 2 KV 组）、vocab 151,936。控制 token（见第 5 节）落在 Qwen2.5 已有的保留 special-token 段（151,600+），基本无需额外扩表。

- 一个值得注意的改动：RoPE 的位置索引被强制用 **float16** 构造（`model.py:773-774`，注释 `### Zhifei fix bug 1:`）——这是相对 litgpt 的一处刻意、且可能有精度损失的偏离，改造时留意。

### 3.2 音频塔（audio tower）
- 编码器 = **Qwen2.5-Omni 的 `thinker.audio_tower`**（Whisper 式 mel 编码器）。加载方式：`Qwen2_5OmniForConditionalGeneration._from_config(cfg).thinker.audio_tower`，再 `load_state_dict`（`extract_online_feature.py:19-22`）。
- **全程冻结**：`.requires_grad_(False).eval()`，在 `torch.no_grad()` 下跑。
- **输入前端**：16kHz → **128 维 log-mel**（`whisper.log_mel_spectrogram(n_mels=128)`）→ tower 两次 stride-2 卷积下采样。1 帧 = 40ms = 640 采样点。
- **输出维度**：tower 编码器隐藏维 1280，经 tower 内部的 `proj` 线性层映射到 `n_embd`。存到 `.pt` 时**已经是 `n_embd` 维**。

### 3.3 融合机制 —— 「embedding 槽位覆盖」
融合**没有** cross-attention、没有 Q-Former、没有门控，就是**把音频特征向量直接覆盖到 token embedding 上**。forward 逐字（`model.py:245-263`）：

```python
x = self.transformer.wte(idx)              # (B, T, n_embd) 词嵌入
# 训练路径：audio_info 是 dict
if isinstance(audio_info, dict):
    if audio_info.get("audio_pos") is not None:
        x = self.fill_in_audio_feature(
            x, batch_size, audio_info["feats_paths"], audio_info["audio_pos"], tasks)
elif torch.is_tensor(audio_info):          # 推理路径：流式单块替换（仅 batch=1）
    x[0, T-(audio_tokens_per_chunk+1):T-1, :] = audio_info
# 之后就是普通自注意力：音频与文本在同一条序列里，无区别对待
for block in self.transformer.h:
    x = block(x, cos, sin, mask, input_pos, input_pos_maxp1)
```

`fill_in_audio_feature`（`model.py:134-169`）按每个样本的 `tasks[i]` 分派：
- `offline`：整块特征一次性放到唯一的音频段 `input_embeddings[i, start:end] = audio_feats[:end-start]`（无维度断言）；
- `online`：按 10 帧一块 `audio_feats[seg*10:(seg+1)*10]` 逐块放，且**有**维度断言校验最后一维 == `n_embd`（`model.py:162-165`，仅在线分支有）。

### 3.4 输出头 & 是否双流
- **只有一个 `lm_head`**（`model.py:49`），输出就是文本词表 logits。**没有任何音频/codec 输出头**。
- 因此它**不是** Moshi/Mini-Omni 那种「文本+声学 token 交错生成」的双流模型。它**消费音频、只吐文本 token（+情绪/控制 token）**，真要出声得靠外部 TTS。
- 所谓「双工/流式」是**回合层面**的：靠 `KEEP_SILENCE` / `TEXT_BEGIN` 在每个音频块后做「听 or 说」决策来实现，见第 5 节。

### 3.5 关于 `audio_adapter` 的一个坑
`wrap_audio_tower.py` 说 `proj` 是作为 `audio_adapter.*` 在主训练里一起训、再事后塞回 tower 的 `proj`。**但当前 `model.py` / `train.py` 里根本没有 `audio_adapter` 这个模块**（全仓库 grep 只在 `wrap_audio_tower.py` 出现）。当前这套 `train.py` 直接 `GPT(model_config)`，并假设磁盘上的特征**已经是 `n_embd` 维**（tower 已 wrap 好）。也就是说：这版公开代码里，**连那个投影线性层都不训**，只训 LM 本体。`wrap_audio_tower.py` / `extract_state_dict.py`（后者剥 `audio_encoder.*`）是给「另一个/更早的、把 adapter 也一起训」的变体用的打包工具。改造时不要被误导。

## 4. 训练数据是怎么构造的

两种数据同一套产出 schema（`input_ids / labels / audio_pos / pt_path_dir`），但建模两种不同的交互范式。

### 4.1 离线数据 `cons_offline_data.py`（单轮、非流式）
整段用户音频编码成**一整块**连续特征，助手回答一次。整段音频上限 20 秒（`MAX_AUDIO_SECONDS=20`）。三种任务变体（取决于哪些块存在）：
- `A_T_T`：音频 + 用户文本 → 助手；
- `A_T`：只有音频 → 助手；
- `T_T`：只有文本 → 助手（纯文本，无 `pt_path_dir`）。

**序列布局**（`cons_offline_data.py:128-137`）：
```
prefix          = [OFFLINE, ENGLISH, SYSTEM(=TEXT_BEGIN), TEXT_BEGIN] + system_ids + [TEXT_END]
audio_block     = [AUDIO_BEGIN] + [AUDIO_PAD]*output_len + [AUDIO_END]     # 有音频时
user_block      = [USER, TEXT_BEGIN] + user_ids + [TEXT_END]              # 有用户文本时
assistant_block = [ASSISTANT, TEXT_BEGIN] + assistant_ids + [TEXT_END]
input_ids = prefix + audio_block + user_block + assistant_block
audio_pos = [(len(prefix)+1, len(prefix)+1+output_len)]   # AUDIO_PAD 那一段
```
监督范围：`labels = [MASK]*(prefix+audio+user 长度) + assistant_block`——**整个助手块（含 `ASSISTANT` 角色 token）都监督**。

### 4.2 在线数据 `cons_online_data.py`（流式、多轮）—— 分 4 步构造
音频被切成固定 **10 帧 = 400ms** 的块（`CHUNK_SIZE=10`）。模型每消费一块就要决策：`KEEP_SILENCE`（继续听）还是开口回复。多轮被拼成一整条长波形，轮与轮之间/首尾塞**随机背景噪声当静音**，让模型学到真实的等待/插话时机。

四步（每步各写一份中间 jsonl，可单独重跑）：
1. **step1 `step1_sample_silence`**：不碰音频。为每轮抽一个前导静音长度 `leading_silence_frames = randint(20,60)` 帧，并选一个噪声文件+偏移；再抽一个尾部静音。（1 帧 = 40ms）
2. **step2 `step2_concat_audio`**：按 16kHz 对齐到整帧，把 `[噪声→轮1音频→噪声→轮2音频→…→尾噪声]` 拼成一条 wav；尾部长度做微调，让总帧数落在 chunk 边界上。
3. **step3 `step3_extract_features`**：对拼好的 wav 跑 Qwen2.5-Omni audio_tower，dump 到 `<features_dir>/<idx>/AudioFeat.pt`，记录 `pt_path_dir`。
4. **step4 `step4_build_tokens`**：铺 token 级流式序列（`input_ids/labels/audio_pos`），写出最终 `tasks:"online"` 的训练 jsonl。

**在线序列布局**（`cons_online_data.py:277-336`）：
```
prefix = [ONLINE, ENGLISH, SYSTEM, TEXT_BEGIN] + system_ids + [TEXT_END]   # 全 mask

# 沉默块（每个非结尾块、以及尾部静音块）：
input : [AUDIO_BEGIN] + [PAD]*10 + [ASSISTANT, KEEP_SILENCE]
labels: [MASK]        + [MASK]*10 + [MASK,      KEEP_SILENCE]      # 只监督 KEEP_SILENCE

# 回复块（每轮的最后一块）：
input : [AUDIO_BEGIN] + [PAD]*10 + [ASSISTANT, TEXT_BEGIN, emotion_tok] + assistant_ids + [TEXT_END]
labels: [MASK]        + [MASK]*10 + [MASK,      TEXT_BEGIN, emotion_tok] + assistant_ids + [TEXT_END]
#                                          ↑注意 ASSISTANT 被 mask，从 TEXT_BEGIN 起才监督
```
- `emotion_tok = EMOTION_TO_ID.get(emotion, NORMAL)`；助手文本若为 `<no need to response>` 则该块退化为沉默块。
- 每块 `audio_pos.append((当前长度+1, +1+10))`，即每个 `AUDIO_BEGIN` 后面那 10 个 `PAD` 槽。
- 第一轮多补一块尾块「给回复留位置」。

> **一个离线 vs 在线的关键差异**（已验证）：
> - **在线**音频占位用 `PAD=151608`，且**没有** `AUDIO_END`；
> - **离线**音频占位用 `AUDIO_PAD=151646`，且**有** `AUDIO_END=151648` 包裹。
> - **在线**回复块里 `ASSISTANT` 角色 token 被 mask；**离线**助手块里 `ASSISTANT` 被监督。这是两条 pipeline 真实存在的不对称，别写混。

### 4.3 DataLoader（`dataloader.py`）
- **懒加载**：扫描 `*.jsonl` 只记录每行的 `(文件, 字节偏移, 长度)`；`__getitem__` 时 seek 到偏移读一行，内存里始终只有一条。读失败返回退化样本 `{input_ids:[PAD], labels:[MASK]}`。
- **取特征**：`torch.load(<pt_path_dir>/AudioFeat.pt, map_location="cpu")`。
- **补齐**：`input_ids` 右侧用 `PAD(151608)` 补、`labels` 用 `MASK(-100)` 补到 `max_seq_len`（=4096）；超长直接硬截断。
- **collate**：只有 `input_ids / labels` stack 成 `(B, 4096)` 张量；`pt_list`（各样本音频特征张量）和 `audio_pos` 因长度不齐，**保持为 Python list**。collate **不做**融合，融合在 model.forward 里做。
- **切分/采样**：按 `eval_data_percentage` 切 train/eval（实际配置 **0.0005**，不是默认 0.1），`random.seed(1337)`；train/eval 都 `shuffle=True`，`num_workers=4`。无按长度分桶（bucketing）。
- `use_turn_map=true` 时启用「随机截断到某一轮」的多轮数据增强；但示例数据没带 `turn_audio_end_map` 字段，所以在示例上这条增强是**空转**的。

## 5. 特殊 token 与序列格式

- **分词器**：litgpt 的 `Tokenizer` 包装类，优先读 `tokenizer.json`（HuggingFace `tokenizers` 后端）。底层是 **Qwen2.5 / Qwen2.5-Omni 分词器**（`TEXT_END=151643=<|endoftext|>`、`TEXT_BEGIN=151644=<|im_start|>` 与 Qwen 约定吻合）。Qwen 无 BOS，数据构造时 `encode()` 不加 bos/eos。
- **音频不是离散 codec token**，而是连续特征经 `PAD` 占位注入（见 3.3）。词表里只有助手**文本**和**控制/决策** token 是真 token。

**控制 token 全表**（`src/audiointeraction/TOKENS.py`，与 `dataset/TOKENS.py` 字节一致；`VOCAB_SHIFT=151600`）：

| 名称 | id | 含义 |
|---|---|---|
| `USER` | 151601 | 用户角色（**在线布局里没用到**，离线用到） |
| `ASSISTANT` | 151602 | 助手角色标记；每个音频块末尾、决策之前出现 |
| `KEEP_SILENCE` | 151605 | ⟨Silent⟩ 决策：继续听、这一步不说话 |
| `PAD` | 151608 | 占位符；每个编码帧一个，被注入的音频 embedding 覆盖（**在线**音频槽） |
| `ONLINE` | 151609 | 任务模式标签：流式（在线样本首 token） |
| `OFFLINE` | 151610 | 任务模式标签：批式 |
| `ENGLISH` | 151611 | 语言标签 |
| `HAPPY/SAD/ANGRY/SURPRISE/NORMAL/URGENT` | 151614–151619 | 情绪标签（`NORMAL=151618` 为默认） |
| `TEXT_END` | 151643 | 文本结束（Qwen `<|endoftext|>`） |
| `TEXT_BEGIN` | 151644 | 文本开始（Qwen `<|im_start|>`） |
| `SYSTEM` | 151644 | 与 `TEXT_BEGIN` 同 id，复用作系统轮标记 |
| `AUDIO_PAD` | 151646 | **离线**音频占位符 |
| `AUDIO_BEGIN` | 151647 | 音频块开始 |
| `AUDIO_END` | 151648 | 音频块结束（**只**离线用） |
| `MASK` | -100 | label 忽略 id（CE ignore_index） |

**一条真实在线样本逐 token 解码**（来自 `examples/online/step4_output.jsonl`，系统提示被简化成单 token `9707`）：
```
idx  id      token          监督 label
0    151609  ONLINE         MASK   ┐
1    151611  ENGLISH        MASK   │ 前缀（全 mask）
2    151644  SYSTEM         MASK   │
3    151644  TEXT_BEGIN     MASK   │
4      9707  <sys 文本>     MASK   │
5    151643  TEXT_END       MASK   ┘
6    151647  AUDIO_BEGIN    MASK   ┐ 块1 = 沉默
7-16 151608  PAD ×10        MASK   │ audio_pos=[7,17]
17   151602  ASSISTANT      MASK   │
18   151605  KEEP_SILENCE   151605 ┘ ← 监督：保持沉默
19   151647  AUDIO_BEGIN    MASK   ┐ 块2 = 沉默
20-29 151608 PAD ×10        MASK   │ audio_pos=[20,30]
30   151602  ASSISTANT      MASK   │
31   151605  KEEP_SILENCE   151605 ┘ ← 监督
32   151647  AUDIO_BEGIN    MASK   ┐ 块3 = 回复
33-42 151608 PAD ×10        MASK   │ audio_pos=[33,43]
43   151602  ASSISTANT      MASK   │
44   151644  TEXT_BEGIN     151644 │ ← 从这里起监督
45   151618  NORMAL(情绪)   151618 │
46      25   <回复文本>     25     │
47   151643  TEXT_END       151643 ┘ ← 收尾
```
读法：模型听过两个 400ms 静音块（都预测 `KEEP_SILENCE`），到第三块决定开口——先 `TEXT_BEGIN`、再一个情绪标签、然后回复文本、最后 `TEXT_END`。**模型被训练去产生的，正好就是所有非 `-100` 的 label 位置。** 流式/双工性完全靠这套「每块一个听/说决策」的节奏实现，没有单独的时间戳/交错 token。

## 6. 训练主循环（`train.py`）

- **入口**：`python -m src.audiointeraction.finetune.train --config src/audiointeraction/finetune/config.yaml`（必须从仓库根以 module 方式跑，因为用了 `from src.audiointeraction...` 绝对导入）。
  > ⚠️ README 和 `config.yaml` 注释里写的 `finetune/full.py` **在仓库里不存在**，真正的入口是 `train.py`。
- **框架**：`L.Fabric(...)` + `fabric.launch(main, ...)`，手写循环。多卡（`devices*num_nodes>1`）→ **FSDP**：`auto_wrap_policy={Block}`、`activation_checkpointing_policy={Block}`（开了激活重算）、`state_dict_type="full"`、`limit_all_gathers=True`、`cpu_offload=False`；单卡 → `"auto"`。
- **建模 & 载权重**：`model = GPT(model_config)`；非续训时 `load_checkpoint(model, ${CHECKPOINT_ROOT}/init.pt)`（`strict=True`）作为种子权重。续训 `resume` 则 `fabric.load` 恢复 model+optimizer+scheduler+计数器。
- **可训练参数**：**全参数微调，什么都不冻**。优化器直接吃 `model.parameters()`，全仓库无 `requires_grad=False`/freeze/LoRA。（第 133-137 行只是把可训练参数名打印出来做诊断。）音频编码器不在 `GPT` 里，自然不训。
- **损失**（`train.py:253-254`）：
  ```python
  logits = model(input_ids, batch["tasks"], batch["batch_size"], audio_info)
  loss = chunked_cross_entropy(logits[..., :-1, :], targets[..., 1:])   # 标准移位
  ```
  `chunked_cross_entropy`（`utils.py:65-96`）内部 `F.cross_entropy(ignore_index=-100, reduction="none")`，再**只除以非 mask token 数**归一。训练 `chunk_size=128`，验证 `chunk_size=0`（整块一次算）。**开源代码里只有一个 CE**（把决策 token 和文本 token 一起算），没有把流式损失单独加权——但这只是论文**加权双损失** `L_LM + λ·L_stream` 在 **λ=1.0** 时的退化写法；论文的真实目标是可调 λ 的双损失 + 四阶段训练，代码并未实现（详见第 8.5 节）。（验证里按数据源分桶只是为了看指标。）
- **梯度累积**：`grad_accum = global_batch/(devices*nodes)/micro_batch = 64/8/2 = 4`。用 `fabric.no_backward_sync` + loss 先除以 4 实现。
- **梯度裁剪**：warmup 期 `max_norm=10.0`，之后 `1.0`（L2）。
- **学习率调度**（`train.py:403-406`）：`LambdaLR(step/200)` 线性 warmup → `CosineAnnealingLR(T_max=max_steps-200)` 余弦退火到 0，`SequentialLR` 在第 200 步拼接。每个**优化器 step** 走一次（不是每个 micro-iter）。
- **精度**：`bf16-mixed`（bf16 不支持则 `16-mixed`）。
- **验证/日志/存档**：
  - 验证：训练前初始验证一次 + 每 400 步 + 训练后最终验证；按数据源分桶报 `val_loss_*` / `val_ppl_*`（eval batch=1）。
  - 日志：TensorBoard，每 3 iter 记一行（loss/step/epoch/lr/iter_time）。
  - 存档：每 1000 步存一次全量 `out_dir/step-*/lit_model.pth`（含 optimizer/scheduler，可续训）；`step≥8000` 后保留 **train-loss 最低的 top-3** `best_step*/`；结束存 `out_dir/final/lit_model.pth`（只含 model）。

## 8.5 ⭐ 论文真实训练法 vs 开源代码（你问的重点）

> 依据：arXiv:2606.05121《Audio Interaction Model》§3.3、附录 B.2、附录 E（Table 11）、§5.4（Table 8），均已从论文 HTML 原文逐字核对。

**先给结论：训练目标是「加权双损失」，而且是「四阶段课程训练」；开源仓库那套单一交叉熵、单阶段全参微调，只是论文配方的一个简化切片，真正的训练主流程（阶段编排 + adapter 训练 + 数据/智能体策展）并没有开源。**

### (1) 真实损失：加权双损失，不是单一 CE

论文 §3.3「Dual-loss Multi-step Streaming Conversion」原式：

```
        1   N  ┌                                                        ┐
  L  =  ─── Σ  │  −log P_θ(t_j | H_j)   +   λ · ( −log P_θ(s_j | H_j) )  │
        N  j=1 └   └──── L_LM ────┘           └──────  L_stream  ──────┘  ┘
```

- **L_LM**：对**文本 token** `t_j` 的交叉熵（标准语言建模）。
- **L_stream**：对**流式控制 token** `s_j ∈ {<silent>, <response>}` 的交叉熵——即「这一块该沉默还是该开口」的决策。
- **λ**：流式目标的相对权重。论文原话大意：这个流式控制 token「构成一个全新的预测目标、是流式交互的核心，因此在标准语言建模目标之外，用一个专门的流式目标去优化它」。
- 附录 B.2：每个位置有两个**互斥**监督目标——`y_stream`（每块一个 silent/response）与 `y_LM`（response 之后的文本 token）；音频编码器位置与指令前缀对两者都 mask。

**λ 的消融（§5.4，Table 8）——这就是你印象里的「权重」：**

| λ | 0.5 | **1.0（采用）** | 2.0 |
|---|---|---|---|
| MMAU ↑ | 58.3 | 58.2 | 57.3 |
| Trigger Acc ↑ | 95.3 | 96.7 | 96.9 |

论文结论：λ 越大触发准确率越高（95.3→96.9），但过大（2.0）会损害理解（MMAU 掉到 57.3），故**采用 λ=1.0** 折中。

**和开源代码的关系（关键）**：开源 `train.py` 里那一句 `chunked_cross_entropy` 把所有未 mask 的 token（既含 silent/response 决策，也含回复文本）一起算平均 NLL——这**恰好等于 λ=1.0 的退化情形**。所以「代码是单一 CE」在描述**代码**时没错，但它 (a) **无法表达 λ≠1**（没有把 L_stream 单拎出来加权的机制，论文能扫 0.5/1/2，代码不能）；(b) 更关键是缺了下面的四阶段。**一句话：开源代码 = 论文双损失在 λ=1 时的特例，且只对应后面的联合阶段。**

### (2) 真实训练是「四阶段课程」，开源代码只有一个阶段

论文附录 E Table 11（逐字）：训练在 **32×H100 80GB**、**bf16 + 梯度检查点 + DeepSpeed ZeRO-2** 上跑约 **10 天**，分四阶段，**每阶段解冻的模块、学习率、batch、步数都不同**：

| | 阶段1 Format | 阶段2 Adapter | 阶段3 Streaming SFT | 阶段4 Instr-following |
|---|---|---|---|---|
| **可训练模块** | 仅 LM head + 新 token 嵌入 | **仅 adapter** | adapter + LM | adapter + LM |
| 学习率 | 1e-4 | 1e-4 | 5e-5 | 1e-5 |
| per-GPU batch | 8 | 8 | 4 | 2 |
| 梯度累积 | 2 | 4 | 8 | 16 |
| 有效 batch | 512 | 1024 | 1024 | 1024 |
| 训练步数 | 5k | 20k | 80k | 15k |
| 数据 | 离线单轮 (A_ins,A_in→T) | 同上格式 | StreamAudio-2M 四大核心能力 | 交错多轮流 + 主动/历史混入 |

四阶段含义（§3.3 原文）：
1. **Format training**：用离线数据教模型目标序列格式 + `<Spe_token>` 的用法。
2. **Adapter training**：**只训 adapter**，把逐块声学表示映射进语言模型空间（格式不变）。
3. **Large-scale streaming SFT**：**联合**训 adapter + LM，覆盖 ASR / S2TT / 对话 / 音频理解四大能力。
4. **Instruction-following fine-tuning**：在交错多轮流上训复杂流式行为（持续辅助、理解感知式介入、主动响应）；**只有阶段4** 才按 Bernoulli 概率混入 history-review（p=0.30）、silent（p=0.40）、proactive（p=0.30）。

公共优化设置（Table 11）：**AdamW（β1=0.9, β2=0.95, ε=1e-8）**、cosine + 线性 warmup（warmup ratio 0.03）、**weight decay 0.01**、max grad norm 1.0、chunk 400ms、最大流长 60 块（24s）。

> ⚠️ 注意 **β2=0.95**（论文）vs 开源代码的 torch 默认 **β2=0.999**（代码只传了 lr）——又一处代码≠论文。

### (3)「comprehension-aware silence」和「history review」是数据层干预，不是新 loss 项

论文 §3.3 + B.2：这两者**不新增 loss 项**，而是由一条**智能体驱动的数据流水线**产出监督数据：
- **history review**：合成「答案依赖至少 3 轮之前内容」的追问，插到序列靠后位置，逼模型做长程上下文检索（仍走 L_LM）。
- **comprehension-aware silence**：大量经智能体审核、判定「不该回应」的静音/非语音片段（依 ProactiveSound-Bench 的 4 条触发准则），边界样本直接丢弃，强化「除非真需要否则保持沉默」（仍走 L_stream）。

### (4) 论文 vs 开源代码 —— 一张对照表

| 维度 | 论文（arXiv:2606.05121） | 开源代码（本仓库） |
|---|---|---|
| 训练目标 | **加权双损失** L_LM + λ·L_stream，λ 可调（=1.0） | 单个 `chunked_cross_entropy`（= λ=1 退化；无 λ 机制） |
| 阶段 | **4 阶段课程**，逐阶段换可训练模块/LR/batch/步数 | **1 阶段**全参微调，不冻结、不分阶段 |
| adapter | 阶段2 单独训 adapter（声学→LM 空间） | `model.py`/`train.py` 里**没有 adapter 模块**；特征假定已投影到 n_embd；只有 `wrap_audio_tower.py`/`extract_state_dict.py` 提到 `audio_adapter.*`（正是这套未开源阶段流程的残迹） |
| 优化器 | AdamW **β2=0.95**, wd 0.01 | AdamW **β2=0.999**（默认）, wd 0.01 |
| 学习率 | 分阶段 1e-4 / 1e-4 / 5e-5 / 1e-5，warmup ratio 0.03 | 单一 2e-4，200 步 warmup |
| batch / 步数 | 有效 batch 512~1024；5k+20k+80k+15k=120k 步 | global 64；epochs=2 |
| 硬件 / 并行 | 32×H100 + DeepSpeed ZeRO-2，~10 天 | config 里 8 卡 + FSDP |
| 最大序列 | 60 块 / 24s | max_seq_length 4096 |
| 静音/历史数据 | 智能体审核 + 阶段4 Bernoulli 混入（0.3/0.4/0.3） | 只有两个 data_source；turn_map 增强在示例上空转 |
| 数据集 | StreamAudio-2M（约 2.6M 条）+ 完整策展流水线 | README 标「Coming」，仓库无 |

**所以对你的三个判断：**
1. **「不是单一 loss」——对。** 论文是 λ 加权的双损失，还专门做了 λ 消融（0.5/1/2）。
2. **「代码和论文不一样」——对。** 代码是 λ=1、单阶段、无 adapter 训练阶段的简化版。
3. **「代码没完全开源」——对。** 阶段编排、adapter 训练阶段、数据/智能体策展、StreamAudio-2M 都没放出；开源的 `train.py` 只能算论文阶段 3/4 的一个可跑骨架。

## 10. 一图总结「它是怎么训的」

> **把冻结的 Qwen2.5-Omni 音频塔当离线特征器，把音频编码成 `n_embd` 维向量存盘；训练时把这些向量直接盖到 `PAD` 占位 token 的 embedding 上，然后对 Qwen2.5-3B 做全参数、下一个 token 交叉熵微调。监督信号只落在两处：① 每个 400ms 音频块之后「继续沉默 `KEEP_SILENCE` / 开口 `TEXT_BEGIN`」的决策 token；② 开口后的「情绪标签 + 回复文本 + `TEXT_END`」。** 8 卡 FSDP、AdamW 2e-4（wd 0.01）、200 步 warmup 接 cosine、bf16、2 个 epoch、序列长 4096。整个「流式听-说」的双工能力，就是靠数据里那套按块排布的 `KEEP_SILENCE`/回复 label 学出来的。
>
> ⚠️ **以上是开源代码的做法。** 论文的真实训练是 **λ 加权双损失 + 四阶段课程**（32×H100 / DeepSpeed ZeRO-2 / 分阶段 1e-4→1e-5 / 约 120k 步），开源版只是它「λ=1、单阶段」的简化切片——详见第 8.5 节。
